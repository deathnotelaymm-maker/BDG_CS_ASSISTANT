# v0.7.0 — Render Business Backend Migration

## What this release fixes

The previous public Guide client converted backend failures into empty arrays. A temporary Worker, Hyperdrive, or Neon failure therefore appeared to customers as “No guide has been published yet.” The previous Worker also performed schema/bootstrap work during normal requests and opened a new PostgreSQL client for each query.

v0.7.0 moves the stateful API to an always-on Render Node service and makes backend failures visible as retryable service errors without deleting or pretending that published content is empty.

## Production stack

- Cloudflare Pages: `bdg-guide-pages`, `bdg-chat-pages`, `bdg-admin-pages`
- Render Web Service: `bdg-ai-help-api-render`
- Render PostgreSQL: `bdg-help-postgres`
- Region: Singapore for both API and database
- Cloudflare R2 bucket: `bdg-ai-guide-images`
- DeepSeek model: `deepseek-chat`

Use paid Render resources for production. The included Blueprint uses a Starter web service and Basic 1 GB PostgreSQL.

## Deployment order

1. Make a Neon database backup.
2. Push this full project to a private GitHub repository.
3. In Render, create a Blueprint from `render.yaml`.
4. Enter every prompted secret and origin value.
5. Temporarily allow your current public IP to access the Render PostgreSQL external URL.
6. Run `MIGRATE-NEON-TO-RENDER-WINDOWS.ps1`.
7. Remove external database access again.
8. Trigger a Render redeploy so the pre-deploy migration runs against the imported data.
9. Verify `/health/ready` and `/health/dependencies`.
10. Deploy all three Cloudflare Pages projects with the Render API URL.
11. Test Admin login, guide editing, image upload, public guides, Smart Match, and AI chat.
12. Keep the old Worker available during the observation period. Disable it only after production verification.

## Required Render environment variables

Required secrets/configuration:

- `ADMIN_EMAIL`
- `ADMIN_PASSWORD` — minimum 12 characters
- `JWT_SECRET` — generated by Blueprint or at least 32 random characters
- `ALLOWED_ORIGINS` — comma-separated exact Cloudflare Pages/custom-domain origins
- `DEEPSEEK_API_KEY`
- `R2_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET_NAME=bdg-ai-guide-images`

`DATABASE_URL` is injected from Render PostgreSQL by the Blueprint.

Example `ALLOWED_ORIGINS`:

```text
https://bdg-guide-pages.pages.dev,https://bdg-chat-pages.pages.dev,https://bdg-admin-pages.pages.dev
```

Add your custom domains too when they are active.

## Render deployment

Generate strong local values:

```powershell
Set-Location "C:\path\to\one-domain-help-ai-admin-v0.7.0-render-business-backend-migration"
.\GENERATE-RENDER-SECRETS-WINDOWS.ps1
```

Push to a private GitHub repository:

```powershell
.\PREPARE-GITHUB-RENDER-V0.7.0-WINDOWS.ps1 -RepositoryUrl "https://github.com/YOUR_ACCOUNT/YOUR_PRIVATE_REPO.git"
```

Then create a Render Blueprint using the repository's `render.yaml`.

## Database cutover

Install PostgreSQL client tools and set the two URLs only in the current PowerShell process:

```powershell
$env:NEON_DATABASE_URL = "YOUR_NEON_CONNECTION_URL"
$env:RENDER_DATABASE_URL = "YOUR_RENDER_EXTERNAL_CONNECTION_URL"
.\MIGRATE-NEON-TO-RENDER-WINDOWS.ps1
Remove-Item Env:NEON_DATABASE_URL
Remove-Item Env:RENDER_DATABASE_URL
```

The script creates:

- a complete Neon source dump;
- a pre-import Render rollback dump;
- a validated restore using `--no-owner` and `--no-privileges`.

Do not paste database URLs into Git, screenshots, or shared logs.

## Cloudflare Pages deployment

After Render reports `0.7.0-render`:

```powershell
.\DEPLOY-CLOUDFLARE-PAGES-V0.7.0-WINDOWS.ps1 -ApiBaseUrl "https://YOUR-RENDER-API-DOMAIN"
```

The script uses `npm ci`, builds each frontend with the Render API URL, and deploys:

- `guide-pro/dist` to `bdg-guide-pages`
- `chat-pro/dist` to `bdg-chat-pages`
- `admin-pro/dist` to `bdg-admin-pages`

## Verification

```powershell
.\VERIFY-V0.7.0-WINDOWS.ps1 -ApiBaseUrl "https://YOUR-RENDER-API-DOMAIN"
```

Expected ready response:

```json
{
  "ok": true,
  "service": "BDG Help Center",
  "version": "0.7.0-render",
  "database": "ok",
  "migration_table": "ok"
}
```

## Cloudflare API custom domain

Recommended production URL:

```text
https://api.yourdomain.com
```

Add that custom domain to the Render web service, create the DNS record requested by Render in Cloudflare, and keep Cloudflare proxy/CDN enabled only after Render domain verification succeeds. Add a Cloudflare Cache Rule for successful public `GET` endpoints if JSON is not cached automatically. Never cache `/admin/*`, `/auth/*`, `/chat`, or upload requests.

## Rollback

Frontend rollback is immediate: redeploy the previous v0.6.8 Cloudflare Pages builds with the old Worker URL.

Backend rollback:

1. Leave the old Worker deployed during migration.
2. Change Page build variables back to the Worker URL and redeploy.
3. Restore the pre-import Render dump only when necessary:

```powershell
$env:RENDER_DATABASE_URL = "YOUR_RENDER_EXTERNAL_CONNECTION_URL"
.\ROLLBACK-RENDER-DATABASE-WINDOWS.ps1 -BackupFile ".\database-backups\render-before-import-TIMESTAMP.dump"
Remove-Item Env:RENDER_DATABASE_URL
```

The original Neon database is not modified by the migration script.
